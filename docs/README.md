# docs Directory Guide

This directory holds every document produced by the AI frontend automation workflow.

## Directory Structure

```
docs/
├── README.md              ← The file you are reading
├── WORKFLOW.md            ← Eight-Step Workflow operation manual
├── ADAPTING.md            ← Cross-Domain Adaptation checklist (read this when forking the framework for backend / data / mobile / DevOps / ...)
├── DECISIONS.md           ← Architecture Decision Records (ADR): context and rationale for major framework decisions
├── tasks/                 ← Task Manifests generated by /plan
│   ├── tasks-user-module-2026-03-30.json
│   └── ...
├── prds/                  ← PRDs generated by /prd
│   ├── _template.md
│   ├── REVIEW.md          ← PRD manual review guide
│   └── login.md
├── bug-reports/           ← Test-side AI test reports (fed to /fix for batch fixing)
│   ├── _template.md
│   ├── README.md          ← Test-side AI prompt snippet + integration notes
│   ├── 2026-04-16-login.md
│   └── screenshots/
└── retrospectives/        ← Read-only observation reports produced by /meta-audit (immutable snapshots)
    ├── README.md
    └── 2026-04-20-meta-audit.md
```

## tasks/ Directory

Holds JSON Task Manifests generated by the `/plan` command. Each file corresponds to the task breakdown of one feature module.

### Naming Convention

```
tasks-[module-name-en]-[date].json
```

### File Contents

```json
{
  "moduleName": "Module name",
  "summary": "Module functionality summary",
  "createdAt": "Creation date",
  "tasks": [
    {
      "taskId": "T001", // Unique task ID
      "type": "api", // Type: api | store | component | hook | page
      "name": "userApi", // File / module name
      "filePath": "workspace/src/...", // Target file path
      "description": "...", // Implementation requirements
      "props": {}, // Props interface definition (components only)
      "dependencies": [], // taskIds of prerequisite tasks
      "reuseComponents": [], // Existing components that can be reused
      "acceptanceCriteria": [], // Acceptance criteria
      "status": "pending" // Status: pending | in-progress | done | blocked
    }
  ],
  "routeConfig": {}, // Route configuration
  "dataFlow": "..." // Data flow description
}
```

### Status Values

| Status      | Meaning                                 |
| ----------- | --------------------------------------- |
| pending     | Not started                             |
| in-progress | In progress                             |
| done        | Completed                               |
| blocked     | Blocked (dependency incomplete or issue)|

### How to Use These Files

```bash
# Reference the Task Manifest while coding
> Please implement starting from T001 based on @docs/tasks/tasks-user-module-2026-03-30.json

# Update task status
> In @docs/tasks/tasks-user-module-2026-03-30.json, change T001's status to done

# Check which tasks are still open
> Read @docs/tasks/tasks-user-module-2026-03-30.json and list every task whose status is not done
```

## prds/ Directory

Holds PRDs — the source of the "requirements → code → tests" Traceability Chain.
**Every `@prd` field in source-code JSDoc references a subheading here as its anchor**, so subheadings must stay stable and clear.

### File Naming

```
docs/prds/<module-name-en>.md   e.g. docs/prds/user-list.md
```

### Authoring Template

See [docs/prds/_template.md](_template.md). Required sections:
- Metadata (module code, owner, status)
- Each feature point under its own H2 heading (`## Search Form`) — the heading is the anchor
- A **Business Rules** section (the source of test assertions; every rule must be testable)
- Field definitions / exception scenarios

### Interplay with `/prd`, `/plan`, and `/test`

```bash
# 0. Conversational requirement → PRD draft (new)
/prd I want to build a login feature
# The AI asks 3-5 key questions, then writes a draft to docs/prds/login.md
# You circle back and fill in the [TBD] items

# 1. Already have a structured PRD
docs/prds/user-list.md  (contains the ## Search Form anchor)

# 2. Break into tasks (the plan command auto-fills prdRef)
/plan @docs/prds/user-list.md

# 3. Coding (source JSDoc receives @prd / @task / @rules)
@prd docs/prds/user-list.md#搜索表单

# 4. Generate tests (the test command reads @prd, pulls rules, and emits one it() per rule)
/test workspace/src/features/list/components/SearchForm.tsx
```

### Anchor Stability

⚠️ Renaming a subheading = changing a URL: every `@prd` reference breaks. If you must rename:
1. Grep the repo for `@prd .*#<old-heading>`
2. Update source-code JSDoc in lockstep
3. Mark "PRD anchor change" in the commit message

## retrospectives/ Directory

The **read-only observation reports** produced by `/meta-audit`, capturing a snapshot of framework health (rule violations / doc drift / broken Traceability Chains / dead references, etc.) at a point in time.

- **Immutable**: once a report is generated, it is never edited; the next audit generates a new report based on the current state, so trends emerge naturally
- **Non-executing**: every change must go through a normal human-reviewed PR; never check off items in the report as "handled"
- **Naming**: `<YYYY-MM-DD>-meta-audit.md`

See [retrospectives/README.md](retrospectives/README.md) for details.

## DECISIONS.md

The Architecture Decision Record (ADR). It captures the context, rationale, and alternatives for every **major design decision** in the framework. Future AIs or new hires reading this will understand "why we did it this way" and avoid mis-editing or retracing dead ends.

See the top of the file for what to record, what not to record, and the template.

## ADAPTING.md

The **Cross-Domain Adaptation checklist**. Although this framework targets frontend, 80% of the machinery is domain-agnostic. If you want to port this skeleton to backend / data / mobile / DevOps / QA / design / product / writing / research / etc., start here:

- What to **copy verbatim** (kernel layer: Eight-Step Workflow skeleton / Traceability Chain / Hard Gates / ADR)
- What to **keep structurally while rewriting contents** (domain layer: the five rule files under `.claude/rules/*`)
- What to **replace wholesale** (domain-specialized layer: the `workspace/` workspace)
- Eight-Step × 9-domain mapping table + eight-step migration checklist
